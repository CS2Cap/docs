title: "CS2Cap Alerts and Webhooks: Setup and Configuration"
description: "Create price and spread alerts that notify you by email, and configure Quant-tier webhooks to receive real-time alert.triggered events in your own systems."
---

CS2Cap gives you two complementary ways to stay on top of market movements. Alerts let you define price and spread conditions on any catalog item — when a condition is met, CS2Cap sends you an email notification. Webhooks (available on Quant) extend this by delivering a signed HTTP payload to your own endpoint whenever an alert fires, so you can build automated reactions without polling. This guide covers both systems from setup to management.

## Alerts

### Alert types

CS2Cap supports three alert kinds:

| Kind | Trigger condition |
|------|-------------------|
| `price_below` | The current best ask falls below your threshold value. |
| `price_above` | The current best ask rises above your threshold value. |
| `spread_exceeds` | The spread `((best_ask - best_bid) / best_ask) * 100` exceeds your threshold percentage. |

### Creating an alert

**Endpoint:** `POST /account/alerts`

```json
{
  "item_id": 156,
  "kind": "price_below",
  "threshold_value": "25.00",
  "threshold_currency": "USD",
  "is_enabled": true
}
```

| Field | Type | Description |
|-------|------|-------------|
| `item_id` | integer | **Required.** Catalog item ID to monitor. |
| `kind` | string | **Required.** One of `price_below`, `price_above`, `spread_exceeds`. |
| `threshold_value` | string | **Required.** Decimal string greater than zero. |
| `threshold_currency` | string | ISO 4217 currency for price alerts. Ignored for `spread_exceeds`. |
| `is_enabled` | boolean | Set to `false` to save the alert in a disabled state. Default: `true`. |

### Alert behavior

- After a successful email delivery, the alert **automatically disables itself**. Re-enable it via `PATCH /account/alerts/{id}` when you want it to fire again.
- A **verified email address** is required before you can enable an alert. Alerts created with `is_enabled: false` do not require a verified email until you enable them.

### Batch alert creation

**Endpoint:** `POST /account/alerts/batch`

Create up to 100 alerts in a single request. The batch counts as one request against your rate limit quota.

```json
{
  "alerts": [
    {
      "item_id": 156,
      "kind": "price_below",
      "threshold_value": "25.00",
      "threshold_currency": "USD"
    },
    {
      "item_id": 298,
      "kind": "price_above",
      "threshold_value": "100.00",
      "threshold_currency": "USD"
    }
  ]
}
```

- Exact duplicate alert payloads within the same request are rejected inline.
- Multiple alerts for the same `item_id` are allowed when the `kind` or `threshold_value` differs.
- If the batch would push you over the enabled-alert cap, the entire request fails with `402`.

### Listing and managing alerts

| Operation | Endpoint |
|-----------|----------|
| List all alerts | `GET /account/alerts` |
| Update threshold or enabled state | `PATCH /account/alerts/{id}` |
| Delete an alert | `DELETE /account/alerts/{id}` |

`GET /account/alerts` accepts a `search` parameter for filtering by exact `item_id` or a case-insensitive item name substring. Results are returned newest first with offset pagination (`limit` up to 200, default 50).

### Alert events

**Endpoint:** `GET /account/alerts/events`

Returns recent alert trigger events and their email delivery attempts, newest first. Use this endpoint to audit which alerts have fired and whether email delivery succeeded.

<Note>
Alert routes are **exempt from monthly quota**. Creating, listing, updating, and deleting alerts does not count against your 1,000 / 500,000 / 1,000,000 monthly request total.
</Note>

## Webhooks

<Warning>
Webhooks are available on the **Quant tier only**. Accounts on Free or Pro will receive a `403` when attempting to create or manage webhooks.
</Warning>

### Creating a webhook

**Endpoint:** `POST /account/webhooks`

```json
{
  "label": "Production alerts",
  "url": "https://hooks.example.com/cs2cap",
  "is_active": true
}
```

Example response:

```json
{
  "webhook": {
    "id": "3b1f1d4c-7b8e-4cc8-bb7b-5d8d0d2a7e11",
    "label": "Production alerts",
    "url": "https://hooks.example.com/cs2cap",
    "secret_last4": "9f2a",
    "is_active": true,
    "last_success_at": null,
    "last_failure_at": null,
    "last_failure_message": null,
    "created_at": "2026-04-12T18:20:11Z",
    "updated_at": "2026-04-12T18:20:11Z"
  },
  "secret": "whsec_live_8f4f4f4c"
}
```

<Warning>
The `secret` field is returned **only once** at creation time. Store it securely (for example in a secrets manager or environment variable). After this response, you can only see the last four characters via `secret_last4`.
</Warning>

The `url` must be a reachable HTTP or HTTPS destination. CS2Cap will POST signed event payloads to this URL whenever an alert fires.

### Webhook event type

CS2Cap delivers one event type to your webhook endpoint:

| Event type | Description |
|------------|-------------|
| `alert.triggered` | An alert condition was met and email delivery was attempted. |

The delivery payload includes the `alert_id`, `item_id`, and `market_hash_name` of the triggered alert.

### Tracking webhook deliveries

**Endpoint:** `GET /account/webhooks/deliveries`

Returns all outbound webhook delivery jobs for your account, newest first. Each delivery record shows the event type, HTTP status of your endpoint's response, attempt count, and timestamps.

To inspect a specific delivery including the full payload and per-attempt history, call `GET /account/webhooks/deliveries/{delivery_id}`.

### Rotating the signing secret

**Endpoint:** `POST /account/webhooks/{id}/rotate-secret`

Generate a new signing secret for an existing webhook. The new secret is returned **once** in the response — update your endpoint's verification logic before rotating to avoid a gap in signature validation.

```json
{
  "webhook": { "...": "..." },
  "secret": "whsec_live_9b1e2c3d"
}
```

### Listing and managing webhooks

| Operation | Endpoint |
|-----------|----------|
| List all webhooks | `GET /account/webhooks` |
| Update label, URL, or active state | `PATCH /account/webhooks/{id}` |
| Delete a webhook | `DELETE /account/webhooks/{id}` |

## Set up your first alert

<Steps>
  <Step title="Verify your email address">
    Before you can enable an alert, your account must have a verified email. If you have not verified yet, complete verification from your account settings — CS2Cap will reject enabled alert creation until this step is done.
  </Step>

  <Step title="Find the item ID">
    Query `GET /v1/items` or `GET /v1/items/metadata` to find the catalog `item_id` for the item you want to monitor. The `item_id` is a stable integer identifier.
  </Step>

  <Step title="Create the alert">
    Call `POST /account/alerts` with your chosen `kind`, `threshold_value`, and `threshold_currency`.

    ```bash
    curl -X POST https://api.cs2c.app/v1/account/alerts \
      -H "Authorization: Bearer <your_api_key>" \
      -H "Content-Type: application/json" \
      -d '{
        "item_id": 156,
        "kind": "price_below",
        "threshold_value": "25.00",
        "threshold_currency": "USD",
        "is_enabled": true
      }'
    ```
  </Step>

  <Step title="Wait for the trigger email">
    When the market condition is met, CS2Cap sends a notification to your verified email address and automatically disables the alert. To have it fire again, re-enable it with `PATCH /account/alerts/{id}` — set `is_enabled` to `true`.
  </Step>

  <Step title="(Quant only) Connect a webhook">
    To receive machine-readable events in addition to email, create a webhook destination with `POST /account/webhooks`. Store the returned secret, then monitor `GET /account/webhooks/deliveries` to confirm delivery.
  </Step>
</Steps>
